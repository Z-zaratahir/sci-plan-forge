WhiteCoat — From Hypothesis to Runnable Experiment Plan
The Problem
Turning a scientific hypothesis into a runnable experiment plan takes weeks of manual work. A pharma company or university sends a brief to a Contract Research Organisation. Scientists then manually scope the protocol, source every reagent with catalog numbers, estimate costs line by line, map out a phased timeline with dependencies, and define how success will be measured. A senior scientist who has done this before can do it in hours. A junior one takes days — and the quality difference is real. A wrong chemical concentration or an unrealistic timeline can send a lab down the wrong path for weeks.
No existing AI tool solves this completely. Tools like Elicit and Consensus do literature search only. GPT wrappers generate a protocol but hallucinate reagent names, invent catalog numbers, and produce no budget or timeline. None of them check their own output for contradictions across sections.
What WhiteCoat Does
WhiteCoat takes a scientific hypothesis typed in plain English and generates a complete, operationally realistic experiment plan — protocol, materials with real catalog numbers, line-item budget, Gantt timeline with critical path, and validation criteria — in under 60 seconds. The output is grounded in real published protocols and real supplier data, not hallucinated. A real scientist could pick it up on Monday and start ordering materials by Friday.
The Core Technical Innovation — PDC Fork-Join Parallel Thread Architecture
The central engineering insight is that the six sections of an experiment plan are mostly independent of each other and can be generated in parallel. Traditional approaches generate them sequentially, which is slow and produces no cross-section consistency checking.
WhiteCoat implements a Parallel and Distributed Computing fork-join model across seven AI threads:
Fork point: Five threads launch simultaneously the moment the hypothesis is submitted — Literature QC, Protocol generation, Budget estimation, Timeline construction, and Validation criteria. These run as concurrent async API calls via Promise.allSettled(), collapsing what would be sequential minutes into the duration of the slowest single thread.
Sequential dependency: The Materials thread is deliberately held back until the Protocol thread completes. This is a data dependency — you cannot list specific reagents until you know what the protocol steps require. This mirrors the PDC concept of a sequential pair inside a parallel set.
Join barrier: The Recombiner is the seventh and final thread. It starts only after all six threads have resolved — this is the synchronisation point. It receives all six outputs simultaneously and performs cross-section coherence checking: equipment mentioned in protocol steps but absent from the materials list, reagent delivery lead times that conflict with the timeline, costs present in the protocol but missing from the budget. This is the PDC join barrier pattern applied to scientific planning.
RAG Pipeline — Tavily-Powered Retrieval Augmented Generation
WhiteCoat uses Tavily's Search API for three distinct retrieval tasks, not just one:
Literature QC: Tavily searches for prior published work matching the hypothesis. Results are passed to the language model which classifies novelty as "not found", "similar work exists", or "exact match found" — with up to three citations. The novelty badge appears in the UI immediately when Thread 0 completes, before the other five threads finish. The scientist sees whether they are breaking new ground before the full plan arrives.
Protocol grounding: Before generating protocol steps, Tavily searches protocols.io and bio-protocol.org for real published protocols matching the hypothesis. The top two results are injected as context into the language model prompt. This is retrieval-augmented generation — steps are grounded in real published methodology, not invented. Each step that draws from a retrieved protocol shows a purple citation chip in the UI with the source URL.
Supplier lookup: Before listing materials, Tavily searches Sigma-Aldrich and Thermo Fisher directly for reagents matching the experiment. Real catalog numbers from search results are used where found. Items not found via search are flagged as "Verify before ordering" in amber — the scientist knows exactly which numbers are confirmed versus estimated.
Critical Path Computation — Kahn's Algorithm on a Task DAG
The timeline section is not a flat list of weeks. The language model returns tasks as a directed acyclic graph with explicit dependency relationships — each task specifies which other tasks must complete before it can start.
WhiteCoat then computes the critical path using Kahn's topological sort algorithm implemented in JavaScript. The algorithm builds an adjacency list and in-degree map from the dependency graph, processes nodes in topological order tracking earliest finish times, and traces back from the latest-finishing task to identify the critical path — the sequence of tasks that determines the minimum possible experiment duration. The Gantt chart renders in D3.js with critical path tasks in black and parallel tasks in grey, with week markers on the x-axis and task names on the y-axis. This is algorithmic post-processing of language model output — the model supplies the dependency data, the algorithm computes what it cannot.
Trust Scorer — Deterministic Rules Plus AI Explanation
Because there is no answer key for an AI-generated experiment plan, WhiteCoat implements a hybrid trust scoring system. A static rules engine checks deterministic conditions first: if the protocol mentions flow cytometry, the materials list must include a flow cytometer. If the protocol mentions PCR, there must be a polymerase or PCR kit. If the protocol mentions ELISA, an ELISA kit must appear in materials. If centrifugation is mentioned, a centrifuge must be listed. These rules run in pure JavaScript with no API call — they are instant and deterministic.
Rule violations are then passed to the language model which writes a plain-English explanation of each violation and suggests how to resolve it. The UI shows a trust badge — green for high confidence, amber for flagged items requiring review, red for low confidence — with expandable warnings below it. The scientist knows at a glance whether the plan is ready to execute or needs review.
Scientist Review Loop — Few-Shot Memory That Improves Over Time
WhiteCoat implements a learning feedback loop without fine-tuning. On the Review page, a scientist can annotate any section of the generated plan — rating it, flagging errors, and writing corrections. Each correction is stored in localStorage as a structured record: experiment type, section name, what was wrong, and what the correct approach is.
When the next plan is generated, WhiteCoat retrieves the top matching corrections by experiment type and injects them as few-shot examples into the language model's system prompt for the relevant thread. The model sees: "A previous expert reviewer flagged this issue in a similar experiment — apply this lesson." No fine-tuning, no retraining — but the output is demonstrably better because expert knowledge is now part of the prompt context. A judge can watch this work live: generate a plan, annotate a section, generate a similar plan, and see the correction reflected in the new output without re-prompting.
Hypothesis Gate — Structured Input Enforcement
WhiteCoat enforces hypothesis quality before generation begins. When the scientist clicks Generate, the hypothesis is first parsed by the language model which extracts four required fields: intervention, measurable outcome with threshold, mechanistic reason, and control condition. If any field is missing or unspecified, the UI shows a targeted inline question — not a generic error. Missing intervention: "What specific treatment or change are you testing?" Missing outcome: "What measurable result would show success, and what is the threshold?" The scientist refines their hypothesis before any expensive generation begins. This prevents the garbage-in-garbage-out failure mode that makes most AI science tools produce plausible-sounding but operationally useless plans.
What Has Never Been Done Before in This Problem Space
Every existing AI science tool does one or two of these things. None of them do all five simultaneously:
First, parallel generation of all six plan sections with explicit dependency management — not sequential, not one big prompt, but a true fork-join architecture where sections generate simultaneously and the recombiner synchronises them.
Second, critical path computation from a real dependency DAG using Kahn's algorithm — the timeline is algorithmically correct, not a flat list of weeks.
Third, a hybrid trust scorer combining deterministic rule checking with AI-generated explanations — the scientist knows which numbers are verified and which need review.
Fourth, RAG grounding across three different data sources for three different sections — literature, protocols, and supplier catalogs — in a single generation pipeline.
Fifth, a scientist review loop where corrections become few-shot examples for future plans of the same experiment type — the system gets better with every review, compounding in value over time.
